# Job Priorities

This page explains how job priorities are generated and how they are used in the buildfarmer workflow.

Priority values are generated by [generate_priorities.py](../../../generate_priorities.py) and saved as [job_priorities.csv](https://github.com/osrf/buildfarm-tools/blob/main/database/scripts/lib/job_priorities.csv).

## Why this exists

The buildfarmer automation uses job priorities to rank issues and focus on the most relevant regressions first.

## Automatic workflow

Job priorities are generated automatically by [Database Fetcher workflow](../../../.github/workflows/databaseFetcher.yml):

1. Runs `python3 generate_priorities.py` in this private repository.
2. Produces a local `job_priorities.csv` file.
3. Moves it to `buildfarm-tools/database/scripts/lib/job_priorities.csv`.
4. Runs Ruby update step:
   * `BuildfarmToolsLib::update_issues_priority`
5. Commits and pushes DB + priorities to `osrf/buildfarm-tools` when scheduled or manually requested with push enabled.

## How generate_priorities.py works

The script loads the jobs matrix from [common.py](../../../common.py) and computes one priority per job.

### Gazebo priorities

Gazebo uses a penalty model:

`Priority = 1.0 - Collection_Penalty - OS_Penalty`

Collection penalties are defined in `GZ_COLLECTION_PENALTY`.
OS penalties are defined in `GZ_OS_PENALTY`.
The final value is rounded to 2 decimals.

### ROS priorities

ROS uses ordered regex rules in `ROS_PRIORITY_RULES`.
The first matching pattern wins.

If no pattern matches, default priority is `1` and the script prints a warning.

## Run locally

From repository root:

```bash
python3 generate_priorities.py
```

This creates:

```text
job_priorities.csv
```

If you need to test full workflow behavior locally, copy the output to the public repository path:

```bash
mv job_priorities.csv buildfarm-tools/database/scripts/lib/job_priorities.csv
```

## Local setup prerequisites

* Python environment with dependencies installed (`requirements.txt`).
* Sibling clone of `buildfarm-tools` if you want to test file move/update into public repo path.

Expected layout:

```text
<parent>/
  buildfarm-tools-private/
  buildfarm-tools/
```

## When to update the script

You may need to update [generate_priorities.py](../../../generate_priorities.py) when:

* A new Gazebo collection is added.
* Platform weights need to change.
* ROS naming conventions change and regex rules no longer match expected jobs.

After any rule update, regenerate the CSV and verify a sample of critical jobs has the expected values.

## Troubleshooting

* **CSV not generated**:
  * Run from repository root and check Python environment is active.
* **Unexpected priority for ROS jobs**:
  * Verify regex order in `ROS_PRIORITY_RULES` (first match wins).
* **Gazebo values look wrong**:
  * Check collection/platform keys match values from [common.py](../../../common.py).
* **Workflow generated file but public repo not updated**:
  * In manual runs, ensure `push-changes` is enabled in workflow inputs.
