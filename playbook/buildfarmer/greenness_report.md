# Greenness Report

This page explains how the monthly greenness report works, what it updates, and how to run it locally.

The report is generated by [`greenness_report.py`](../../../greenness_report.py).

## What it does

Given a date range, the script reads build data from the local buildfarmer database and:

1. Generates an HTML report with per-project and per-release greenness metrics.
2. Includes release summaries (current period vs previous month).
3. Shows status charts and per-job greenness tables.
4. Updates monthly values in the **Buildfarm Issue Triage** spreadsheet:
	 * `ROS Greenness History`
	 * `Gazebo Greenness History`

Output HTML filename format:

```text
greenness_buildfarm_metrics_report_<from>_to_<until>.html
```

## Data sources

The script reads from the SQLite database managed by buildfarmer tools using [`lib/database.py`](../../../lib/database.py), mainly from:

* `server_status` (projects/releases/jobs metadata)
* `build_status` (build results)

For each release, it computes success/unstable/failure/aborted percentages and total build counts in the requested time window.

## Spreadsheet update logic

When updating Google Sheets, the script:

1. Authenticates with service-account credentials (This is a buildfarmer's own credential).
2. Opens spreadsheet **Buildfarm Issue Triage**.
3. Resolves ROS and Gazebo projects from database project names.
4. Looks up each release column by header name (normalized match).
5. Finds the row for the target month in the `Month` column.
6. Writes the greenness value as percentage (for example, `92.41%`).

## Automation workflow

The monthly workflow is defined in [`.github/workflows/monthlyGreennessReport.yml`](../../../.github/workflows/monthlyGreennessReport.yml):

1. Runs monthly on day 1 (`0 12 1 * *`) and can also be triggered manually.
2. Installs dependencies and clones the public repository.
3. Generates last-month report by calling:

```bash
python3 greenness_report.py -f "$from_date" -u "$until_date"
```

4. Moves the generated report to `buildfarm-tools/reports/greenness_reports/<year>/`.
5. Pushes to `osrf/buildfarm-tools` when scheduled or when manually requested.

Then [`buildfarm-tools/.github/workflows/greenness-reports-deploy.yml`](../../.github/workflows/greenness-reports-deploy.yml) deploys report pages to GitHub Pages.

## Run locally

### Prerequisites

* Python 3.11 recommended.
* Local clone of this private repository.
* Local clone of public [`buildfarm-tools`](https://github.com/osrf/buildfarm-tools) as a sibling folder (required by [`lib/database.py`](../../../lib/database.py)).
* Valid local database in `buildfarm-tools/database/`.
* Spreadsheet service-account credentials in [`credentials.json`](../../../credentials.json).

### 1) Setup environment

From the repository root:

```bash
python3 -m venv .venv
source .venv/bin/activate
python3 -m pip install --upgrade pip
python3 -m pip install -r requirements.txt
```

### 2) Ensure database and repository layout

Expected layout:

```text
<parent>/
	buildfarm-tools-private/
	buildfarm-tools/
```

If `buildfarm-tools` is missing, clone it next to this repository:

```bash
cd ..
git clone https://github.com/osrf/buildfarm-tools
```

If your DB is stale, update it first (see [Buildfarmer triage tools](./buildfarmer_triage_tools.md#database-fetching-scripts)).

### 3) Configure spreadsheet credentials

Create a Google service-account key and save it as [`credentials.json`](../../../credentials.json), then share the target spreadsheet with that service account.

You can follow the credential creation steps in [Update Issue Spreadsheet](./update_issue_spreadsheet.md#create-spreadsheet-credentials).

### 4) Run the report

With explicit date range:

```bash
python3 greenness_report.py -f 2026-02-01 -u 2026-03-01
```

Or with open-ended end date (defaults to current date):

```bash
python3 greenness_report.py -f 2026-02-01
```

## Troubleshooting

* **Database not found / empty results**:
	* Verify sibling `buildfarm-tools/database` exists and has data.
	* Refresh database before generating report.
* **Spreadsheet auth errors**:
	* Check [`credentials.json`](../../../credentials.json) exists and is valid.
	* Ensure spreadsheet is shared with the service-account email.
* **Worksheet not found**:
	* Confirm tabs `ROS Greenness History` and `Gazebo Greenness History` exist.
* **Release columns not found**:
	* Verify spreadsheet headers match release names used in the database.
